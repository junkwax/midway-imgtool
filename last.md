Based on the codebase of the modern C++ port of midway-imgtool, several features from the original loadimg pipeline are already present. Since midway-imgtool is a GUI-driven application, these exist as UI workflows and environmental features rather than CLI flags.

Here is the breakdown of what has been added and what we can add:

✅ Features already added (Implemented)
/I - use IMGDIR for source images: Fully implemented. IT/it.c checks the IMGDIR environment variable to set the default directory for loading .img files, matching the original DOS tool behavior.
/T= - directory for table files (.tbl/.asm/.glo): Implemented via the GUI's export features. You can write these files using WriteTblFromMarked (for .tbl files) and WriteAnilstFromMarked (for assembly animation lists) from marked sprites in the UI.
/A - append mode: Implemented. The GUI includes an "Append IMG" file dialog mode (FileDialogMode::AppendImg) that merges the contents of an incoming .img file into the current workspace.
/B - bpp from palette size: Handled conceptually within the Tools → Verify LOAD2 Packing feature. load2_verify.cpp accurately simulates LOAD2's ppp fallback logic (jumping the bits-per-pixel if the palette exceeds 1 << ppp colors).
🚀 Features we can add (Not yet implemented)
/F= (generate .irw) &/X(do not generate IRW): midway-imgtool can currently simulate and verify LOAD2 packing to warn you about alignment drift, but it does not actually compile the final .irw raw binary blobs. Adding an IRW builder would be a major and highly useful feature.
/D= (specify directory for .lod file): midway-imgtool currently operates on .img files directly. Adding a .lod manifest parser to batch-load libraries and apply overarching directives (like PPP>) would streamline the workflow.
/E(dual-banked image memory / ED adjustment): Not currently supported. We would need to implement the specific Midway ED adjustment logic for how images are mapped across dual memory banks.
**/P(pad to 4-bit boundary) &/L(align to 16-bit boundary):**These hardware-specific padding/alignment constraints aren't explicitly enforced during export right now. We could add these as toggleable export settings or integrate them into the LOAD2 verifier.
/3 (limit scales to 3): Scaling limits aren't currently enforced by the tool; this could be added as an optional validation check.
/V(verbose): While mostly irrelevant for the GUI, if we ever introduce a headless CLI batch mode for midway-imgtool, a verbose logging flag would be useful.

Verbose Logging Mode (/V)

How we'd do it: We can introduce a g_verbose global variable, toggleable via the UI (or a command-line arg if we add CLI parsing), and sprinkle printf statements in img_io.cpp.
LOD Manifest Parser (/D)
